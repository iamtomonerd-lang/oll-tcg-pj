# TCGエンジン 層構造ガイド

このプロジェクトは、複数のTCGタイトルを同じ基盤で実装するための汎用ゲームエンジンです。

## 階層構造

コードは「層」で責任範囲を分離しています。

**重要**: 層番号は上下関係ではなく、責任の大きさを表します。

```
層0（最大の責任）
  ↓
層1
  ↓
層2
  ↓
層3（最小の責任）
```

数字が小さいほど責任が大きく、抽象的です。
数字が大きいほど責任が小さく、具体的です。

## 各層の役割

### 層0：ゲームに存在するもの

**位置**: `src/層0_ゲームに存在するもの/`

**責任**: ゲームに存在する基本的なエンティティ（プレイヤー、カード、ゾーン、ゲーム状態）の「構造」を定義する

**重要な原則**:
- 「何を持つことができるか」のみを定義
- 「どのような値が入るか」は定義しない
- 「どう動くか」は定義しない
- TCG全体に共通する基本概念のみ
- 特定ゲームのルールは含まない

**ファイル**:
- `zone.py` - ゾーンの基本構造
- `card.py` - カードの基本構造
- `player.py` - プレイヤーの基本構造
- `game_state.py` - ゲーム状態の基本構造

### 層1：共通処理

**位置**: `src/層1_共通処理/`

**責任**: 層0で定義されたエンティティを操作するための共通的な処理を実装

**特性**:
- 層0のエンティティを利用する
- 複数のTCGで同じ処理を共有可能
- 特定ゲームのルール判定は含まない

**予定されるモジュール**:
- ゾーン間のカード移動処理
- ターン進行の管理
- プレイヤーアクション実行の枠組み
- ゲーム状態変更の管理

### 層2：TCG個別ルール

**位置**: `src/層2_TCG個別ルール/`

**責任**: 特定のTCGタイトルに固有のゲームルールを定義

**構成**:
- `sample_tcg/` - サンプルTCGの実装（検証用）
- `other_tcg/` - 他のTCG実装（将来追加）

**このレベルで初めて登場する概念**:
- 「手札は最大7枚」という制限
- 「メインフェイズ」「エンドフェイズ」の定義
- 「ライフが0以下で敗北」という勝利条件
- ゾーン設定（「手札」「場」など）

### 層3：カード定義

**位置**: `src/層3_カード定義/`

**責任**: 個別のカード情報を定義

**構成**:
- `sample_tcg/` - サンプルTCGのカード定義
- `other_tcg/` - 他のTCGのカード定義（将来追加）

**役割**: データベース的な役割
- カード名、効果テキスト
- 数値情報（攻撃力、コストなど）
- 分類情報（種族、属性など）

## 依存ルール

### 層間依存の原則

- **下位層（小さい番号）は上位層（大きい番号）を知らない**
  - 層0は層1,2,3を参照しない
  - 層1は層2,3を参照しない

- **上位層（大きい番号）は下位層（小さい番号）を使用する**
  - 層1は層0を使用
  - 層2は層0,1を使用
  - 層3は層0,1,2を使用

- **下位層の変更は上位層に波及する（許容）**
  - 層0の変更は全層に影響
  - 層1の変更は層2,3に影響

- **上位層は下位層を変更してはいけない（禁止）**
  - 層2が層0,1を変更することは許されない
  - 層3が層0,1,2を変更することは許されない

## 複数TCG対応の例

別のTCGを追加する場合：

```
src/
├── 層0_ゲームに存在するもの/  ← 共有（変更なし）
├── 層1_共通処理/               ← 共有（変更なし）
├── 層2_TCG個別ルール/
│   ├── sample_tcg/            ← サンプルTCG
│   └── other_tcg/             ← 別のTCG（新規）
└── 層3_カード定義/
    ├── sample_tcg/            ← サンプルTCGのカード
    └── other_tcg/             ← 別のTCGのカード（新規）
```

層2と層3でのみ差分管理し、層0,1は共有します。

## コード品質の原則

### 用語の選択

**優先する名前**: 日本語のカードゲーム用語
- 手札、山札、トラッシュ
- 召喚、使用、破壊
- 攻撃、防御
- 効果

**避ける名前**: プログラミング専門家向け
- entity、executeAction
- resolveStateTransition
- processObject

### ドキュメント

各層のモジュールには、以下を明確にするコメントを記載します：
- その層の責任は何か
- 何を定義し、何を定義しないか
- 上位層との関係

## 開発スケジュール

1. **層0の完成** - ゲーム基本エンティティの構造定義
2. **層1の実装** - 共通処理の実装
3. **層2の実装** - サンプルTCGルール
4. **層3の実装** - サンプルTCGカード定義
5. **検証** - サンプルTCGで動作確認
6. **拡張** - 他のTCG実装、AI対応など

## AI対応への配慮

各層は以下を考慮して設計されています：

- **層0**: AIが状態を解析可能な構造
- **層1**: AIが有効なアクション一覧を取得可能
- **層2,3**: AIが状態遷移を予測可能

エンジンとAI判断は分離され、AIが直接ゲーム状態を変更することはありません。
