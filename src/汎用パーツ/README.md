# 汎用パーツ

複数のシステムで再利用可能な汎用的な部品の集合。

**位置づけ：層外共有資産**

---

## アーキテクチャ内での役割

```
        汎用パーツ
        ↙ ↓ ↓ ↘
     層0 層1 層2 ゲーム側
```

汎用パーツは**階層ではなく**、すべての層が共有して利用できる資産。

---

## 設計方針

### 汎用パーツが担当するもの

✅ 数値管理  
✅ プレイヤー情報管理  
✅ ゾーン機能  
✅ カード情報管理  
✅ 条件判定  
✅ データ管理  
✅ その他の再利用可能な部品  

### 汎用パーツが含まないもの

❌ ゲーム固有ルール  
❌ 特定ゲームの名前  
❌ 勝敗条件判定（具体的な）  
❌ ターンシステム（具体的な）  
❌ 効果実装  

---

## 層別の責務区分

```
層0：概念・型定義
  └─ 何が存在するか

層1：実行処理
  └─ 実際にどう処理するか

層2：処理管理・監視
  └─ 処理の順番を管理し、状態変化を通知

汎用パーツ：共有資産
  └─ すべての層が利用できる再利用可能な部品

ゲーム側（層3+）：意味付け・ルール定義
  └─ パーツに名前・ルール・具体的な用途を付与
```

---

## 再利用可能性の原則

汎用パーツは**複数の利用場所**を想定して設計される。

### 数値管理パーツの例

```
汎用パーツが提供：
  数値管理インターフェース
    ├─ 現在値
    ├─ 最大値
    └─ 変更・参照機能

利用先1（ゲーム側）:
  └─ ライフ管理として使用

利用先2（UI側）:
  └─ 表示用データとして使用

利用先3（サーバー側）:
  └─ 同期用データとして使用
```

---

## 責任範囲の明確化

### 許可される設計

```typescript
// 汎用パーツ：再利用可能な部品
interface 数値 {
  現在値: number;
  最大値: number;
  変更(新しい値: number): void;
}

// ゲーム側で意味付け
// 「このパーツをライフとして使用」
```

### 禁止される設計

```typescript
// ❌ ゲーム固有ルールを含む
interface ライフ {
  現在値: number;
  敗北判定(): boolean;  // ゲーム固有ルール
}
```

## 実装判断基準

新しい機能を追加するとき、以下のツリーで判断：

```
Q1: 処理を管理するものか？
  ├─ はい → 層2
  └─ いいえ
      ↓
Q2: 実際に処理を行うものか？
  ├─ はい → 層1
  └─ いいえ
      ↓
Q3: ゲームごとに意味が変わるか？
  ├─ はい → ゲーム側（層3+）
  └─ いいえ
      ↓
Q4: 複数の場所で再利用できるか？
  ├─ はい → 汎用パーツ
  └─ いいえ → 検討し直す
```

## 用語の定義

- **層外共有資産**：特定の層に属さず、すべての層が利用できる資産
- **再利用可能**：複数のゲーム、複数のシステム、複数の用途で利用できる
- **ゲーム非依存**：特定のゲームのルールを含まない
