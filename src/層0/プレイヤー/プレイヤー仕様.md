# 0層 プレイヤー構造仕様

## 定義の目的

プレイヤーという存在が持つことができる基本情報のみを定義する。

具体的なゲームルール、TCG固有の数値（「ライフ」「マナ」など）や状態、分類は上位層で定義する。

## プレイヤーが持つ情報

```
プレイヤー
├ 名前
│   └ 型: 文字列
│
├ 状態
│   ├ 所持数
│   │   └ 型: 数字
│   └ 内容
│       └ 型: 所持数分の文字を配列で保持
│
├ 数値情報
│   ├ 所持数
│   │   └ 型: 数字
│   └ 内容
│       └ 型: 所持数分の数字を配列で保持
│
├ 分類情報
│   ├ 所持数
│   │   └ 型: 数字
│   └ 内容
│       └ 型: 所持数分の文字を配列で保持
│
└ 権限
    └ 型: 文字列
```

## 各属性の説明

### 名前
- **型**: 文字列
- **説明**: プレイヤーの識別名
- **例**: 「Player 1」「太郎」「AI対戦相手」
- **補足**: プレイヤー個別の名前は層3で定義される

### 状態
- **構造**:
  - 所持数：数字（何種類の状態を持つか）
  - 内容：所持数分の文字を配列で保持
- **説明**: プレイヤーが複数の状態を持つことができる
- **例**:
  - 所持数: 2
  - 内容: ["待機中", "ターン実行中"]
  - （意味は層2以上で決定。例：[ゲーム状態, ターン段階]）
- **補足**: 具体的な状態名（「横向き」「中毒」など）は層2以上で定義される

### 数値情報
- **構造**:
  - 所持数：数字（何種類の数値を保持するか）
  - 内容：所持数分の数字を配列で保持
- **説明**: プレイヤーが複数の数値を持つことができる
- **例**:
  - 所持数: 3
  - 内容: [20, 5, 10]
  - （意味は層2以上で決定。例：[ライフ, マナ, エネルギー]）
- **補足**: 数値の具体的な意味は層2以上で定義される

### 分類情報
- **構造**:
  - 所持数：数字（何種類の分類を保持するか）
  - 内容：所持数分の文字を配列で保持
- **説明**: プレイヤーが複数の分類タグを持つことができる
- **例**:
  - 所持数: 2
  - 内容: ["赤チーム", "初心者"]
  - （意味は層2以上で決定。例：[チーム, ランク]）
- **補足**: 分類の具体的な意味は層2以上で定義される

### 権限
- **型**: 文字列
- **説明**: プレイヤーの権限レベルまたは役割
- **例**: 「プレイヤー」「AI」「オブザーバー」「admin」
- **補足**: 具体的な権限の種類と意味は層2以上で定義される

## 重要な設計原則

### プレイヤーとゾーンの責務分離

**プレイヤーはゾーンを持たない**

- ゾーンの所有者管理は「ゾーン」が行う
- プレイヤーからゾーンを参照する構造にはしない
- これにより、プレイヤーの責務を明確に保つ

**責務の方向：**
```
ゾーン → プレイヤー（ゾーンがプレイヤーの所有者情報を参照）

プレイヤー ↛ ゾーン（プレイヤーはゾーンを参照しない）
```

### 可変性の特徴

#### 状態の可変性
- 異なるプレイヤーで異なる数の状態を持つことが可能
- 例：
  - プレイヤーA：1個の状態 ["ターン開始"]
  - プレイヤーB：3個の状態 ["ターン開始", "バフ状態", "制限状態"]
  - プレイヤーC：0個の状態 []

#### 数値情報の可変性
- 異なるプレイヤーで異なる数の数値を持つことが可能
- 例：
  - プレイヤーA：2個の数値 [20, 5]
  - プレイヤーB：3個の数値 [25, 3, 10]
  - プレイヤーC：1個の数値 [15]

#### 分類情報の可変性
- 異なるプレイヤーで異なる数の分類を持つことが可能
- 例：
  - プレイヤーA：1個の分類 ["人間プレイヤー"]
  - プレイヤーB：2個の分類 ["AI", "難易度:難"]
  - プレイヤーC：0個の分類 []

## 層0が定義しないもの

- **具体的な状態の種類**
  - 例：「横向き状態」「中毒状態」は層2で定義

- **具体的な数値の意味**
  - 例：「ライフポイント」「マナポイント」は層2で定義

- **具体的な分類の意味**
  - 例：「赤チーム」「難易度レベル」は層2で定義

- **具体的な権限の種類**
  - 例：「管理者権限」「ゲストユーザー」は層2で定義

- **ゾーン管理**
  - プレイヤーがゾーンを参照・管理することはない

## 設計の利点

1. **責務の明確化**: プレイヤーとゾーンの役割が明確に分離
2. **柔軟性**: 異なるゲームシステムに対応可能
3. **拡張性**: 新しいゲーム要素の追加が容易
4. **保守性**: 層間の責任が明確に分離されている
5. **一貫性**: カード・ゾーンと同じ構造パターンを使用
